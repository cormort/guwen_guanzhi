# 古文觀止文章品質標準

## 核心原則

**不要為了寫而寫。** 每一段分析都必須對讀者理解這篇文章有實際幫助。如果一個章節拿掉之後讀者不會損失任何理解，那它不應該存在。

具體而言：
- **起承轉合表**：只在文章確實有清晰的層次推進時才寫。短篇銘文（如〈陋室銘〉）或對話體不必硬套四段式。
- **修辭手法表**：只列這篇文章裡**有辨識度的**手法，不要把「對比」「排比」這種每篇都能硬湊的通用修辭當填充。問自己：這個手法是不是這篇文章**值得注意**的特色？
- **佳句賞析**：選的句子必須是這篇文章的**文眼或名句**，不是隨便挑一句加上「語氣有力」四個字就交差。賞析要說出**為什麼這句好、好在哪裡**，不能只是把原文換個說法重述一遍。
- **白話改編**：如果原文已經很白話（如明清小品），不需要再翻譯一次。標注「原文已近白話，不另改寫」即可。
- **背景資料**：只補充讀者**不知道就讀不懂原文**的背景。不要寫成百科全書式的作者生平流水帳。

## 判斷方法

對每一段補寫的內容，問三個問題：
1. **刪掉這段，讀者會少理解什麼？** 答不出來就不該寫。
2. **這段是不是在說這篇文章獨有的東西？** 如果換成另一篇文章也成立，就是廢話。
3. **讀者讀完這段，能不能用自己的話說出一個新的觀點？** 如果只是「嗯，我知道了」而沒有「原來如此」，就是為了寫而寫。

## 自動檢查項（quality-audit.py）

| 檢查項 | 權重 | 說明 |
|--------|------|------|
| 座標定位表 | 3 | 出處、體裁、時代、核心命題 |
| 時代背景段 | 2 | 讀懂原文所需的最少背景 |
| 關鍵人物段 | 2 | 文中出現的人物，不知道就讀不懂的那些 |
| 寫作意圖段 | 2 | 作者為什麼寫、寫給誰 |
| 一句話總結 | 2 | 一句話抓住全文核心論點 |
| 起承轉合表 | 3 | 文章結構層次（不適用時可標注跳過） |
| 修辭手法表 | 3 | 本文有辨識度的修辭（不是湊數） |
| 佳句賞析 ≥2 | 3 | 文眼或名句，附深度賞析 |
| 白話改編段 | 2 | 近白話原文可標注跳過 |
| 標題無 emoji | 1 | h2 標題統一不用 emoji |
| 分析段 ≥60行 | 2 | 分析內容有實質深度 |

**注意：** 自動檢查只能抓「有沒有」，不能抓「好不好」。通過所有自動檢查不代表品質合格——核心原則（不要為了寫而寫）由人工或 agent 審閱把關。

## Agent 評審指令

### 前置條件

1. 專案根目錄包含 `articles/*.html`（267 篇文章）
2. 評審結果寫入 `quality-reviews/` 目錄（不存在則建立）
3. 每篇文章產出一個 `quality-reviews/{檔名}.json`
4. 已存在的 JSON 跳過不重跑

### 逐篇評審流程

對每一篇 `articles/*.html`：

**步驟 1：讀取文章全文**

讀取 HTML 檔案，理解文章的背景段（`bg-section`）和分析段（`analysis-section`）。

**步驟 2：逐項檢查 12 項**

| # | id | 檢查項 | 怎麼判 |
|---|-----|--------|--------|
| 1 | `bg_coord` | 座標定位表 | 是否有表格列出出處、體裁、時代、核心命題 |
| 2 | `bg_era` | 時代背景段 | 是否提供讀懂原文的最少背景（不是百科流水帳） |
| 3 | `bg_people` | 關鍵人物段 | 是否涵蓋不知道就讀不懂的人物 |
| 4 | `bg_intent` | 寫作意圖段 | 是否說明作者為什麼寫、寫給誰 |
| 5 | `bg_summary` | 一句話總結 | 是否用一句話精準抓住全文核心論點 |
| 6 | `an_structure` | 起承轉合表 | 結構分析是否合理。短篇銘文、對話體可判 `not_applicable` |
| 7 | `an_rhetoric` | 修辭手法表 | 是否列出本文**有辨識度的**修辭。通用修辭（對比、排比）湊數算 fail |
| 8 | `an_quotes` | 佳句賞析 | 是否選出 ≥2 句文眼/名句並深度賞析。只翻譯不賞析算 fail |
| 9 | `an_vernacular` | 白話改編段 | 是否提供白話版。原文已近白話可判 `not_applicable` |
| 10 | `fmt_emoji` | 標題無 emoji | h2 標題是否統一不用 emoji |
| 11 | `fmt_depth` | 分析段深度 | 分析段是否有實質深度（扣除原文引用後是否還有獨立觀點） |
| 12 | `no_padding` | 不為寫而寫 | **最重要**。見下方「判定指引」 |

**步驟 3：每項給 status + comment**

- `pass`：合格。comment 說好在哪裡，或可改進之處
- `fail`：缺失或品質不足。comment 具體指出問題，引用文章內容
- `not_applicable`：此文不適用此項。comment 說明原因

**步驟 4：給整體等第**

- A：≥10 項 pass
- B：8-9 項 pass
- C：6-7 項 pass
- D：<6 項 pass

**步驟 5：寫入 JSON**

  "title": "鄭伯克段于鄢",
  "author": "左丘明",
  "overall_grade": "B",

  "checks": [

      "status": "pass",

    },
    {
      "id": "bg_era",
      "name": "時代背景段",
      "status": "pass",
      "comment": "介紹周室東遷與鄭國崛起背景，對理解莊公野心有實質幫助。"

### 「不為寫而寫」判定指引

以下情況都算 **fail**：

| 問題類型 | 具體表現 | 範例 |
|----------|----------|------|
| 重複段落 | 白話改編與分段大意說同一件事 | 分段大意翻譯了一次，白話改編又翻譯一次 |
| 翻譯冒充賞析 | 佳句賞析只是白話翻譯 | 「此句意為天下最難的事……」而不說美在哪裡 |
| 通用修辭湊數 | 修辭手法表列出每篇都有的手法 | 「對比」「排比」不是本文特色就不該列 |
| 百科流水帳 | 背景段寫了跟這篇文章無關的生平 | 作者從出生寫到死亡但跟本文主旨無關 |
| 空洞評語 | 評語刪掉不影響理解 | 「此乃千古名篇」「值得細讀」 |

**判定方法：刪掉這段，讀者會少理解什麼？答不出來就是 fail。**

### 彙整報告

所有文章評審完成後，讀取 `quality-reviews/*.json`，產出以下統計：

1. **等第分佈**：A / B / C / D 各幾篇
2. **各項 fail 排名**：哪個檢查項最多文章不及格
3. **問題文章清單**：C/D 級文章列表，附各篇缺項和評語摘要

將統計寫入 `quality-reviews/_summary.json`。
